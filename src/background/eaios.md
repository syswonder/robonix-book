# EAIOS 架构背景

Robonix 是 EAIOS（Embodied AI Operating System）架构的参考实现。EAIOS 由 CCF 泛在操作系统开放社区在 2026 年发布的《具身智能操作系统技术白皮书》中提出，核心思路是在机器人系统层面建立统一的抽象体系，让 AI 模型与硬件之间实现软硬解耦。

白皮书原文：[gitlink.org.cn/zone/uos/source/292](https://gitlink.org.cn/zone/uos/source/292)

## 四级抽象

EAIOS 定义了四级抽象体系，自底向上分别是原语（Primitive）、服务（Service）、技能（Skill）和任务（Task）。

原语是对硬件能力的标准化封装，每一种传感器或执行器暴露一组确定的接口——例如相机提供 RGB 图像流，底盘提供速度控制和导航命令。不同厂商的同类硬件实现同一套原语接口，上层调用者无需关心底层差异。

服务是对算法能力的封装。VLM 推理、语义地图查询、路径规划等通用算法以服务形式注册到系统中，任何消费者都可以通过统一的发现和协商机制使用它们。

技能将一组原语和服务调用组合为可复用的行为单元。一个"巡检"技能可能需要底盘导航、相机感知和 VLM 推理的协同，而技能本身作为一个整体对外暴露，可以被更高层的任务编排直接调用。

任务处于最顶层，负责将用户的自然语言指令分解为技能调用序列，并编排执行。

## 三空间模型

白皮书将具身智能系统划分为任务空间、配置空间和操作空间三个维度。任务空间描述"做什么"——用户的意图和目标；配置空间描述"用什么"——哪些硬件和算法资源可用；操作空间描述"怎么做"——实际的控制信号和感知数据流转。Robonix 的控制平面对应配置空间中的资源注册与发现，数据面对应操作空间中的实际通信。

## Robonix 中的对应

Robonix 目前实现的组件与 EAIOS 四级抽象的对应关系：

| EAIOS 层 | Robonix 实现 |
|----------|-------------|
| 原语 | `robonix/prm/*` namespace 下的接口定义及 provider 实现（如 `tiago_bridge`） |
| 服务 | `robonix/sys/*` namespace 下的系统服务（如 VLM `chat` 接口） |
| 技能 | SKILL.md 格式 + `QueryAllSkills` RPC + Agent 技能发现 |
| 任务 | `robonix-agent` 的 ReAct 循环（VLM 驱动的任务规划与执行） |

控制平面（`robonix-server`）负责节点注册、接口声明和通道协商，是配置空间的核心；数据面通信通过 gRPC、MCP、ROS 2、共享内存等传输方式完成，对应操作空间。
