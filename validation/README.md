# Page evidence gate

`page-evidence.yaml` is the review ledger for the active handbook under `docs/`.
It is deliberately kept outside `docs/`, so evidence-review notes are not
published as a handbook page.

The ledger pins every page to `ROBONIX_SOURCE_REVISION`, names the relevant
Robonix `dev-next` implementation, API, protocol, configuration, or test paths,
and classifies every fenced block. A page addition, fence addition, fence body
change, line move, missing source path, or missing classification fails the
normal gate.

## Scope and exclusions

All Markdown and MDX files under `docs/` are active unless listed in the
ledger's `exclusions`. The only current exclusions are:

- `docs/reference/contracts.md`, generated by `rbnx docs` from the pinned
  contract catalog;
- `docs/reference/idl.md`, generated by `rbnx docs` from the pinned IDL tree.

Their generator paths and source revision remain recorded in the ledger.
Images and other non-Markdown assets are outside this gate and are checked by
the existing asset check. Build output, archived copies, and versioned or
translated generated trees are not under `docs/` and are therefore outside the
active-page inventory.

## Classifications and evidence

- `runnable` means the page presents the block as a command, file, or snippet
  intended to run in the stated context.
- `safety-gated` means the block can activate hardware or participates in a
  motion-capable workflow. Its profile must record the last non-motion check
  and the required real-hardware acceptance conditions and evidence.
- `illustrative`, `partial`, `output`, and `reference` are explicitly
  non-standalone presentations. They cannot silently inherit runnable status.

Evidence is attached to the exact block group, not inherited from a profile.
Runnable groups record one of four requirements: `command-contract`,
`runtime-smoke`, `end-to-end`, or `staged-non-motion`. A passed syntax run
never satisfies those requirements. A `passed` claim must name a passed run
whose scope is strong enough for the requirement; failed and blocked runs stay
in the ledger as regression or prerequisite evidence without making a block
green.

When the source pin advances, a failed or blocked run from the old revision may
remain with `historical: true` and its exact old commit. Historical evidence can
never satisfy a block: passed and pending runs must be rerun at the current pin,
and a historical run cannot be relabeled passed. This keeps reproduced
regressions visible without letting stale success evidence make current pages
green.

Safety-gated groups can use `staged-passed` when an actual integration or
hardware run proves the last non-motion boundary. Their real-hardware status
remains separate and may stay `pending`, with the required operator, emergency
stop, limit, observed-motion, and shutdown evidence retained in the profile.
This avoids claiming a hardware result from syntax or from a generic mock.
A full safety `passed` status additionally requires a named, current-pin,
passed run whose scope is explicitly `hardware`; prose status alone cannot
promote a staged non-motion check to real-hardware acceptance.

The normal gate enforces the page inventory, source provenance, classifications,
fingerprints, and evidence-record consistency while allowing honestly pending
or blocked execution. Strict mode additionally fails every runnable group that
is not passed and every safety group that has not reached at least a truthful
staged non-motion pass.

## Commands

Run the structural gate:

```bash
npm run check:page-evidence
```

Validate every source path against an exact, clean checkout of the pinned Git
tree:

```bash
node scripts/check-page-evidence.mjs \
  --source /absolute/path/to/robonix \
  --require-source
```

List exact pending, failed, or blocked execution gaps:

```bash
npm run check:page-evidence:strict
```

Inspect the current fence inventory:

```bash
npm run page-evidence:inventory
```

After documentation edits, generate a review candidate without overwriting the
ledger:

```bash
npm run --silent page-evidence:refresh > /tmp/page-evidence.yaml
diff -u validation/page-evidence.yaml /tmp/page-evidence.yaml
```

The refresh maps unchanged fences by content fingerprint and preserves their
block-specific validation records. Once the ledger is tracked, the committed
ledger and committed pages are the refresh baseline; working-tree edits to a
stored fingerprint cannot carry old evidence onto changed content. On the
initial untracked bootstrap the command emits an explicit trust-boundary
warning. New, edited, duplicate-content ambiguous, or otherwise ambiguous
fences are emitted with profile `needs-review`; because that profile is
intentionally undefined, copying an unreviewed candidate into the ledger keeps
the gate red.
Review each such block, select a real profile, bind only evidence that actually
covers the new content, and then update the ledger with the reviewed candidate.
Runs marked `binds_to_docs` also carry a whole-fence-inventory digest, so a
documentation edit invalidates the old syntax/build snapshot until those
commands are rerun and the new digest is recorded.
