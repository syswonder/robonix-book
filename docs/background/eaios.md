# EAIOS 架构背景


Robonix 是具身智能操作系统（Embodied AI Operating System，EAIOS）架构的参考实现。EAIOS 由中国计算机学会（China Computer Federation，CCF）泛在操作系统开放社区于 2026 年发布的《具身智能操作系统技术白皮书》中提出，其核心目标是在机器人系统层面建立统一的抽象体系，实现 AI 模型与硬件之间的软硬解耦。

白皮书原文：[gitlink.org.cn/zone/uos/source/292](https://gitlink.org.cn/zone/uos/source/292)

## 四级抽象

EAIOS 采用"原语–服务–技能–任务"四层抽象体系（白皮书 §3.2），实现从高层语义目标到底层硬件控制的系统化分层建模。

原语（Primitive）构成硬件抽象层，定义"抽象机器人"的标准化接口。白皮书将原语分为两类：感知原语（Perception Primitive）负责采集原始观测数据（相机 RGB/深度、IMU、关节编码器、力觉触觉等）；动作原语（Action Primitive）定义确定性的执行指令（底盘速度控制、机械臂关节控制、夹爪开合等）。不同厂商的同类硬件实现同一套原语接口，上层应用与模型通过原语与抽象机器人交互，无需感知底层差异。

服务（Service）是由操作系统统一注册、调度与管理的功能组件，为任务的决策与执行提供全链路能力支撑。白皮书定义的典型服务包括：语义地图、空间地图、任务规划、方案推演、决策、数据采集、物体识别、校准、记忆、结果反馈，以及人机交互与协同服务。开发者遵循统一的接口规范开发并接入服务，系统在运行时根据任务需求动态完成服务的实例化与编排。

技能（Skill）封装具有特定语义的、可复用的行为操作序列，连接高层任务与底层原语。白皮书将技能分为两种形式：基本技能以独立执行单元注册到系统中（如预训练的视觉-语言-动作模型（Vision-Language-Action，VLA）、ROS 2 执行算法等），具有固定运行实现的静态技能；机器人任务描述语言（Robot Task Description Language，RTDL）技能则是在任务执行过程中由系统动态生成的控制流程描述，在运行时由 RTDL 解释器调度与组合执行。当某类任务的执行流程在多次运行中被验证为稳定可靠时，系统可将其固化为可复用的技能存入技能库。

任务（Task）是具身智能系统运行的顶层逻辑单元，由用户或系统自主提出，具备明确的目标状态与终止条件。系统使用 RTDL 对任务方案进行形式化描述，经由完整的生命周期管理：任务规划（生成 RTDL 方案）→ 方案推演（世界模型验证可行性与安全性）→ 决策执行（调度器分配资源、协调技能与原语）→ 结果反馈（评估执行效果并反馈至系统闭环）。

## Robonix 中的对应

Robonix 作为 EAIOS 架构的参考实现，其组件与四级抽象的对应关系如下：

| EAIOS 层 | 白皮书定义 | Robonix 对应 |
|----------|---------|------------|
| 原语 | 硬件抽象接口：感知原语（相机、IMU 等）与动作原语（底盘、机械臂等） | `robonix/primitive/*` 下的能力约定与能力提供方实现（如差速底盘、相机、激光雷达原语，一个物理设备对应一个软件包） |
| 服务 | 操作系统统一注册与调度的功能组件：感知、认知、互联、控制等 | `robonix/service/*` 下的可替换服务；当前源码树提供记忆、语音和声纹参考软件包，Webots 部署另从外部仓库引入建图和导航。Scene、Pilot 等系统组件使用 `robonix/system/*` 接口，不属于 `robonix/service/*` 能力约定命名空间 |
| 技能 | 可复用行为单元：基本技能（预训练 VLA 等）与 RTDL 技能（运行时动态生成） | 进程形态的基本技能以软件包部署，并以技能提供方向 Atlas 注册 MCP 能力；Pilot 动态生成 RTDL 树，Executor 负责解释与执行。将稳定 RTDL 图固化为可复用技能的库管理尚在规划中 |
| 任务 | 用户请求的结构化表达，经由规划→推演→决策→执行→反馈的完整生命周期 | Liaison 把客户端输入构造为任务（`Task`）发给 Pilot；Pilot 把任务分解为 RTDL 方案，并根据 Executor 返回的节点和方案结果继续规划；世界模型方案推演仍在规划中 |

Liaison 对应白皮书 §3.4.3 的人机交互与协同服务，负责接入文本和语音交互、统一用户身份元数据、执行当前用户准入检查，并把任务提交给 Pilot。Atlas 是 Robonix 的运行时控制平面，负责能力提供方注册、能力约定声明、发现和通道记账；它能注册和发现技能提供方，但不保存或发布技能软件包。软件包发布与检索由 Robonix 软件包目录（Package Catalog）负责。

## 系统作用域与非目标

Robonix 聚焦于具身智能系统的运行时框架。以下事项不在当前作用域内。

### 启动与上电

Robonix 的“启动”指功能启动：`rbnx boot` 根据机器人部署清单编排内置系统组件、原语、服务与技能软件包；`rbnx start` 只负责启动单个软件包。硬件级上电管理（例如主控板向关节控制板发送电源指令）不在当前作用域内，应由机器人部署和厂商安全系统在启动 Robonix 前完成。

### 模型运行与训练

Robonix 当前负责模型推理与执行，即以技能形态运行预训练模型（VLA、强化学习策略、视觉语言模型等）。训练、数据采集和模型微调工具链不属于当前运行时接口；训练产物需要由技能软件包封装后部署。

### 控制面定位

Atlas 仅承担运行时提供方注册、能力发现、能力约定目录、生命周期状态与通道记账，不承担软件包或技能库发布，不参与数据面转发，也不负责部署进程编排。数据面通信由能力提供方与消费者在取得端点后直连完成；Atlas 的心跳机制只用于标记和清理超时注册记录。
